# 00_PRD — 재난안전MVP 제품 요구사항 문서

> 본 문서는 공모전(제품·서비스 개발 부문) 제출용 **작동형 웹 MVP**의 제품 요구사항을 정의한다.
> 완성형 상용 서비스가 아니라, **심사위원에게 핵심 시연 흐름을 안정적으로 보여주는 시제품**을 목표로 한다.
> 최종 갱신: **2026-06-16** (전면 리팩토링 R-series 반영 — 실서비스형 흐름: 로그인 시작·재난유형 자동분류·읽기형 대응계획·공유/발송)

---

## 1. 제품 개요

- **프로젝트명**: 재난안전MVP
- **한 줄 설명**: 지자체 재난문자를 유아교육기관의 실제 운영상황에 맞춰 **원장·담임교사·통학버스 담당자·조리사/급식담당자·보건담당자별 실행 대응계획**과 **학부모 안내문**으로 변환하고, 역할별 담당자에게 **공유(링크·인쇄·문자·알림톡)**하는 AI 기반 재난대응 웹 MVP.
- **시작 시나리오**: 지자체가 재난문자를 발송 → 유아교육기관 원장이 수신 → 본 서비스에 **기관 계정으로 로그인**하는 시점부터 흐름이 시작된다.
- **지원 재난유형**: **폭염(heatwave) · 집중호우(heavy_rain) · 감염병(infection)** 3종 구현.
- **지원 역할**: **원장(director) · 담임교사(homeroom_teacher) · 통학버스담당자(bus_manager) · 조리사/급식담당자(cook_or_food_service) · 보건담당자(health_manager)** 5종.
- **배경**: 재난문자는 일반 시민 대상의 포괄적 안내에 머문다. 유아교육기관(어린이집·유치원)은 영유아라는 고위험군을 다수 보호하지만, 현장에서 "그래서 지금 무엇을 해야 하는가"를 역할별로 즉시 실행 가능한 행동으로 번역하는 도구가 없다. 본 제품은 그 **번역(Translation) 갭**을 메운다.

## 2. 문제 정의

| # | 문제 | 현장 영향 |
|---|---|---|
| P1 | 재난문자가 일반 시민 기준 → 유아기관 현장 행동으로 미번역 | 원장·교사가 매번 자체 판단, 누락·지연 발생 |
| P2 | 역할별(원장/담임/통학버스/조리·급식/보건) 책임이 분화되지 않음 | "누가 무엇을" 불명확, 중복·공백 |
| P3 | 학부모 안내문 작성 부담 + 과도한 공포 표현 위험 | 안내 지연 또는 불안 조장 |
| P4 | 사후기록 작성 부담 | 조치 사실 입증·개선 누적 어려움 |
| P5 | 취약 유아(온열·호흡기·이동지원·특별지원 등) 정보가 개인정보 우려로 활용 곤란 | 위험 대응이 일반화됨 |
| P6 | 폭염 외 집중호우·감염병 상황에서 유아기관 전용 행동지침 부재 | 재난유형별 맞춤 대응 불가 |

→ **해결 방향**: 재난문자 + 기관정보 + 현재 운영상황(집계값 기반)을 입력하면, AI가 재난유형을 자동 분류하고 **역할별 대응계획 / 학부모 안내문**을 JSON으로 생성한다. 결과는 역할별로 **공유(링크·인쇄·문자·알림톡)**한다. 유아 개인식별정보는 수집·전송하지 않으며, 공유·발송 목적의 교직원 업무 연락처만 수신동의 전제로 저장한다.

## 3. 핵심 사용자

| 역할 | 주요 니즈 | MVP 진입점 |
|---|---|---|
| **유아교육기관 원장** | 로그인 후 재난문자 입력·대응계획 생성·역할별 공유 총괄 | 기관 로그인 → 대응계획 생성 |
| **담임교사** | 보육실 단위 즉시 실행 행동 | 공유 링크/문자로 받은 담임 대응계획 |
| **통학버스 담당자** | 차량 안전(탑승 전/운행 중/하차) 행동 | 공유 링크/문자로 받은 통학버스 대응계획 |
| **조리사/급식담당자** | 급식·식재료 안전, 조리실 위험 대응 | 공유 링크/문자로 받은 조리 대응계획 |
| **보건담당자** | 유증상 관찰·분리·보호자 연락·공식기관 안내 | 공유 링크/문자로 받은 보건 대응계획 |
| **지자체 관리자** | 관할 기관(1,000개+) 현황·우선순위 모니터링 | 관리자 대시보드(검색·페이지네이션) |

> 원장이 **기관 계정(등록번호+PIN)으로 로그인**해 대응계획을 생성하고, 나머지 역할 담당자는 원장이 공유한 **역할별 링크/문자/알림톡**으로 자신의 대응계획만 열람한다(별도 로그인 불필요).

## 4. MVP 목표

공모전 제출 전까지 **작동 가능한 웹 MVP**를 완성한다. 다음 흐름이 끊김 없이 시연되어야 한다(3개 시연 경로 모두).

```
기관 로그인 (등록번호 + PIN)
  → [미등록 시] 기관 등록: 어린이집정보공개포털 API 검색·자동채움
                 + 재난안전 추가정보(프로필) + 역할별 담당자 연락처(수신동의)
  → 재난문자 입력 (실시간 조회 또는 원문 붙여넣기) → AI가 재난유형 자동 분류
  → 현재 기관 상황 선택(최대 3개, 재난유형별 상황 버튼) + 당일 운영 인원(선택)
  → AI 맞춤 대응계획 생성(JSON)
  → 원장·담임교사·통학버스·조리사/급식담당자·보건담당자별 대응계획(읽기) + 학부모 안내문
  → 역할별 공유: 링크 복사 / 출력·PDF / 문자(SMS) / 카카오 알림톡
  → (관리자) 대시보드에서 3유형별 현황·기관 검색 확인
```

> 변경점(R-series): 시작점이 **역할 선택 → 기관 로그인**, 재난유형은 **수동 선택 → AI 자동 분류**, 결과는 **체크리스트 토글 → 읽기형 대응계획**, **사후기록 기능 제거**, **공유/발송 기능 신규**. 화면상 샘플 데이터 선택 UI는 제거(백엔드 fallback은 유지).

## 5. 기능 요구사항

### FR-1. 기관 로그인 (간편 인증)
- 기관 **등록번호(login_id) + PIN(4~8자리)** 로그인. PIN은 해시 저장(scrypt), 세션은 HMAC 서명 쿠키(`ds_session`, httpOnly, 12h).
- 보호경로(`/plan`·`/account`·`/institutions`)는 `proxy.ts`가 미인증 시 `/login`으로 가드.
- 미등록 기관은 FR-2 등록으로 진입. 등록 직후 자동 로그인.
- 샘플 모드(`USE_SAMPLE_FALLBACK=true`)에서는 임의 입력으로 샘플 기관 세션을 발급해 시연 무중단.

### FR-2. 기관 등록 (어린이집정보공개포털 API 연동 필수)
- **포털 API 검색·자동채움**: 기관명/유형으로 어린이집정보공개포털(어린이집)·유치원알리미(유치원) 조회 → 후보 선택 시 기본정보 자동 입력. 키 미설정 시 예시 후보 fallback.
  - 활용 필드: `stcode`(식별코드), `crname`, `crtypename`, `craddr`, `la/lo`(좌표), `crcapat`(정원)·`crchcnt`(현원), 연령별 아동수·반수, 교직원 현황(`em_cnt_*`), `crcargbname`(통학차량). API 원본은 `institutions.api_raw`에 보존.
  - 파생값: `infant_total_count`(만0~2세+영아혼합), `preschool_total_count`(만3~5세+유아혼합), `special_support_count_api`(특수장애 아동수, 집계값만).
  - 역할 자동활성화 근거: 간호사/간호조무사>0→보건, 영양사/조리원>0→조리·급식, 통학차량 운영→통학버스, 특수교사/특수장애아동>0→특별지원 강화(법적 단정 금지).
- 자동채움 후 수정 시 `child_count_source='user_corrected'`로 기록.
- **로그인 정보 설정**: 등록번호 + PIN.
- **재난안전 추가정보(프로필)**: 등록 직후 재난유형별 프로필 입력으로 이어짐(FR-4).
- **역할별 담당자 연락처**: 공유·발송을 위한 연락처 등록(FR-11).
- **급식·보건 인력 정보** (`staff_profile` JSONB): 1회 급식 인원, 급식·보건 인력 보유 여부·수·유형 등(집계값·boolean만).

### FR-3. 재난유형 자동 분류 (신규 — 수동 선택 폐지)
- 재난문자 입력 시 **AI/키워드로 재난유형을 자동 분류**(폭염/집중호우/감염병). 1차 키워드 분류 → 미분류 시 AI 보조 1콜.
- 분류 결과를 저장하고 이후 프로필·상황 버튼·AI policy block 분기에 활용.
- 자동 분류 실패('other') 시에만 사용자가 3유형 중 수동 선택.

### FR-4. 재난유형별 대응 프로필 (6.3 확장)

**폭염 프로필 (`disaster_specific`):**
온열취약 유아 수, 호흡기 주의 유아 수, 이동지원 필요 유아 수, 특별지원 필요 유아 수, 냉방기 정상 여부, 실내 대체활동 공간 여부, 정수기/물 공급 여부, 차량 내부 온도계, 하원 대기 장소 유형.

**집중호우 프로필 (`disaster_specific`):**
저지대 위치 여부, 인근 하천·배수로·급경사지 인접 여부, 지하공간 보유 여부, 1층 출입구 구조, 하원 대기 장소, 실외 놀이터 위치, 대체 실내 대기공간 확보, 비상연락망 구비 여부, 침수·정전 시 대피공간, 이동지원 필요 유아 수.

**감염병 프로필 (`disaster_specific`):**
반별 유아 수(집계값), 영아반 여부, 특별지원 유아 수, 보건실/분리대기 공간, 손소독제·마스크·소독용품 보유, 보호자 연락 방식(앱/문자/전화/게시판), 감염병 대응 매뉴얼 보유, 등원중지 안내 템플릿 보유.

- **공통**: 체온계 보유 여부, 구급함 보유 여부, 실내 대체활동 공간 여부.
- **유아 이름·진단명·보호자 연락처 저장 금지. 민감정보는 집계값(숫자) 또는 boolean만.**

### FR-5. 재난문자 입력 (화면 샘플 선택 제거)
- 입력 방식 2가지: **실시간 조회**(행안부 긴급재난문자 API) / **원문 붙여넣기**. 화면상 샘플 선택 UI는 제거.
- 입력 즉시 FR-3 자동 분류 수행.
- 백엔드 fallback(API 키 미설정/실패 시 샘플 반환)은 유지하되, 사용자가 "샘플을 고르는" UI는 노출하지 않는다.

### FR-6. 현재 상황 선택 (6.5 확장)
- 버튼형 UI, **최대 3개 선택**. + **당일 운영 인원**(당일 재원·등원 유아수/출근 교직원수, 선택 집계값) 입력 → AI가 등록 정원 대신 실제 규모로 우선순위 조정.
- 상황 버튼은 재난유형별로 다르게 표시(registry 기반 동적 로드):
  - **폭염**: 실외놀이 전·실외놀이 중·산책·현장학습 예정·급식 전후·낮잠 시간·하원 준비·통학버스 탑승 전·냉방 이상·온열증상 의심·조리실 온도 상승·급식 식재료 보관 확인 등 11개.
  - **집중호우**: 실외놀이 전·실외놀이 중·하원 준비·통학버스 운행 전·통학버스 운행 중·지하공간 사용 중·현장학습 예정·기관 주변 침수 우려·보호자 인계 지연 가능·정전·누수 발생·조리실 누수·급식 지연·낙상·불안 반응 유아 등 12개+.
  - **감염병**: 유증상/발열 유아·구토·설사·호흡기 증상·같은 반 유사증상 반복·보호자 연락 필요·등원중지 안내 필요·교실 소독·급식·화장실 위생·교직원 유증상·조리종사자 유증상·급식·식기 위생 강화 등 12개+.

### FR-7. AI 대응계획 생성 (읽기형 — 체크리스트 토글 폐지)
AI는 다음을 **JSON**으로 생성: 재난유형, 재난문자 요약, 대응 우선순위, 반영된 근거 정보, 부족한 정보, **역할별 행동 배열(`role_based_actions[]`, 5역할 동적)**, 학부모 안내문, 응급 연락 안내, 공식기관 지시 우선 안내문, 안전 고지문.
- 결과 화면은 역할별 탭에서 **읽기 전용 대응계획**으로 표시(체크박스 토글·진행률·`checklist_items` 저장 폐지).
- **JSON 파싱 실패 시 재난유형별 샘플 결과 fallback 표시.**
- 비활성 역할은 `actions: []` 또는 "해당 없음"으로 처리(탭에서 안내 표시).
- 사후기록(`after_action_draft`) 기능은 제거(스키마 optional로만 잔존, 화면 미노출).

### FR-8. 학부모 안내문 (6.7)
- 결과 화면 학부모 탭 + 공유 링크(`/share/[token]/parent`). 과도한 공포 표현 금지. 기관이 이미 조치 중이라는 안정감 제공. 재난유형에 맞는 구체 행동 포함.

### FR-9. 공유 / 발송 (신규)
- 대응계획을 역할별로 전달:
  - **공유 링크**: 역할별 공개 URL `/share/[token]/[role]`(인증 불필요, `action_requests.share_token` 기반).
  - **출력/PDF**: 인쇄 친화 레이아웃(`@media print`).
  - **문자(SMS) / 카카오 알림톡**: `/api/plan/[id]/notify` → 등록된 역할별 연락처·수신동의 기준 발송. 알림톡 우선 → SMS fallback → 키 미설정/샘플모드 시 발송 시뮬레이션.
- 발송 본문에는 유아 개인정보를 포함하지 않는다(역할명 + 공유 링크만).

### FR-10. 관리자 대시보드 (1,000개+ 기관 대비)
- 통계: 등록 기관 수, 오늘 생성 대응계획 수, 고우선순위 기관 수(24h), **재난유형별 3종 건수**, **최근 30일 활성 기관(계획 생성) 수**.
- 기관 목록: **검색 + 페이지네이션 + count**(전체 로드 폐지, `/api/admin/institutions`). 기관 상세 보기.

### FR-11. 역할별 담당자 연락처 (신규)
- 5역할별 담당자 **이름·휴대폰·이메일 + 수신동의(문자/알림톡/공유링크) + 사용여부** 등록(`institution_staff_contacts`).
- 교직원 업무 연락처는 공유·발송 목적에 한해 저장(수신동의 전제). **학부모 연락처는 저장 금지.** anon SELECT 정책 없음(service_role 전용).

## 6. 비기능 요구사항

| 구분 | 요구사항 |
|---|---|
| **시연 안정성** | 네트워크/외부 API/AI 실패 시에도 **샘플 fallback**으로 전 흐름 동작. 데모 중단 금지. 3유형 × 5역할 모든 경로 보장. |
| **응답 시간** | AI 생성 목표 10초 이내, 초과 시 로딩 표시 + 타임아웃 후 fallback. |
| **개인정보·안전** | 아래 §10 원칙 준수. AI API에 개인식별정보 미전송. |
| **접근성/모바일** | 현장 모바일 사용 가정. 큰 터치 타깃(min-h-[44px]), 버튼형 UI, 반응형. |
| **신뢰성 표기** | 결과 화면에 안전 고지 문구 고정 노출. |
| **유지보수** | registry 패턴으로 재난유형 추가 시 코드 분기 없이 설정 객체만 추가. |

### 개인정보 및 안전 원칙 (필수)
- 유아 이름 저장 금지 / 진단명 저장 금지 / 약물명 저장 금지 / 보호자(학부모) 연락처 저장 금지.
- 취약 유아 정보는 숫자 집계값만 저장.
- AI API에는 개인식별정보를 전달하지 않음(화이트리스트 필드만 직렬화). **교직원 연락처도 AI 입력에 미포함.**
- **예외(2026-06-16, D-006)**: 대응계획 공유·발송 목적에 한해 **교직원 업무 연락처**(역할별 담당자)는 수신동의와 함께 `institution_staff_contacts`에 저장 허용(anon 미노출). 학부모 연락처는 계속 저장 금지.
- AI는 의료 진단을 하지 않음. 공식기관·보건당국·의료기관 지시 우선 안내 필수.
- AI는 공식기관 지시보다 우선한다고 표현하지 않음.
- 결과 화면 고정 문구: **"공식 재난문자와 기관 입력정보를 바탕으로 한 대응지원 정보이며, 위급상황에서는 공식기관 지시와 119 안내를 우선합니다."**

### 감염병 추가 안전 원칙
- 질병명 확정·진단 금지. "확진"은 사용자 명시 입력 시에만 사용.
- 보건당국·의료기관·기관 매뉴얼 우선 안내 필수.
- 낙인·차별 우려 표현 금지. 증상 집계값만 사용.

### 역할 추천 원칙
- 기관 재원 수·학급 수 기반 역할 활성화 추천은 **"확인이 필요합니다"** 수준에 한정.
- "배치 의무입니다" 등 법적 단정 표현 금지.

## 7. 제외 범위 (MVP 미구현)

모바일 앱 / 스마트폰 재난문자 자동 수신 / 통학버스 노선 위험분석 / 이미지 분석 / 동영상 분석 / 유아별 건강정보 관리 / 실제 지자체 전체 기관 권한체계 / IoT 센서 연동.

추가 재난유형(**대설·화재·미세먼지** 등)은 **확장 기능 문서**로만 정리. (집중호우·감염병은 구현 완료.)

**어린이집정보공개포털·유치원알리미 API는 연동 완료**(키 미설정 시 예시 fallback). 침수위험지도 / 하천정보 / 대피소 API는 MVP 제외.

**카카오 알림톡·문자(SMS) 발송은 구조 구현 완료**(`/api/plan/[id]/notify`). 단 알림톡 실발송은 사업자 채널·발신프로필·템플릿 승인이 필요하므로, 키·승인 확보 전에는 발송 시뮬레이션(fallback)으로 동작.

## 8. 성공 기준

### 8.1 핵심 5원칙 (최우선)
1. 시연 흐름이 끊기지 않을 것 (3개 시연 경로 모두).
2. 샘플 데이터 fallback이 있을 것 (3유형 × 5역할).
3. AI 결과가 JSON으로 안정적으로 표시될 것.
4. 개인정보를 과도하게 수집하지 않을 것.
5. 재난문자·기관정보·현재상황이 AI 결과에 근거로 반영될 것.

### 8.2 측정 가능 기준
- [x] 신규 사용자가 로그인/등록→대응계획→공유까지 **막힘 없이 1회 완주** 가능 (경로 A/B/C).
- [x] 외부 API/AI 차단 환경에서도 fallback으로 전 화면 정상 동작 (smoke-test 30/30 PASS, 샘플모드).
- [x] AI 출력이 정의된 JSON 스키마(04 문서)를 100% 충족, 파싱 실패 시 자동 fallback.
- [x] DB·AI 전송 데이터에 유아 개인식별정보 0건(스키마·프롬프트 차원에서 원천 차단).
- [x] 결과 화면에 "재난문자 요약 + 반영 근거"가 항상 표시되어 입력→결과 연결이 드러남.

### 8.3 심사 데모 체크리스트 (3개 시연 경로)
- [x] **기관 로그인 → 재난문자 입력(자동분류)** 시연.
- [x] **경로 A(폭염)**: 폭염 재난문자 1건으로 5역할 읽기형 대응계획 생성 시연.
- [x] **경로 B(집중호우)**: 하원 준비·통학버스 운행 전·지하공간 사용 중 상황으로 5역할 대응계획 생성 시연.
- [x] **경로 C(감염병)**: 발열 유아 발생 상황 입력으로 5역할 대응계획 + 학부모 안내문 생성 시연.
- [x] 역할별 공유 링크 복사 / 인쇄 / 발송(시뮬레이션) 동작 시연.
- [x] 관리자 대시보드에서 3유형별 우선순위·기관 검색 확인 시연.
- [x] 안전 고지 문구 노출 확인 (3유형 전부).

## 9. 용어
- **대응계획(action plan)**: 하나의 재난문자·상황 입력에 대해 AI가 생성한 결과물 묶음(역할별 읽기형 행동 + 학부모 안내문).
- **공유 토큰(share_token)**: 대응계획을 인증 없이 역할별로 열람할 수 있는 공개 링크 식별자.
- **집계값**: 개인을 식별하지 않는 숫자 합계(예: 온열취약 유아 수, 발열 유아 수, 당일 재원 수).
- **fallback**: 외부 의존(AI/API/DB) 실패 시 미리 준비된 샘플 데이터로 흐름을 유지하는 메커니즘(화면 노출 없이 백엔드에서만 동작).
- **registry**: 재난유형별 상황 버튼·프로필 필드·AI policy block 설정 객체(`lib/disaster/registry.ts`). 유형 추가 시 이 파일만 수정.
- **role_based_actions**: AI 출력의 역할별 행동 배열. 5역할 × 3유형 조합으로 생성.
