# 00_PRD — 재난안전MVP 제품 요구사항 문서

> 본 문서는 공모전(제품·서비스 개발 부문) 제출용 **작동형 웹 MVP**의 제품 요구사항을 정의한다.
> 완성형 상용 서비스가 아니라, **심사위원에게 핵심 시연 흐름을 안정적으로 보여주는 시제품**을 목표로 한다.

---

## 1. 제품 개요

- **프로젝트명**: 재난안전MVP
- **한 줄 설명**: 지자체 폭염 재난문자를 유아교육기관의 실제 운영상황에 맞춰 **원장·담임교사·통학버스 담당자별 실행 체크리스트**, **학부모 안내문**, **사후기록**으로 변환하는 AI 기반 재난대응 웹 MVP.
- **1차 재난유형**: **폭염**만 구현. (집중호우/대설/화재/감염병/미세먼지는 확장 기능으로 문서에만 정리)
- **배경**: 폭염 재난문자는 일반 시민 대상의 포괄적 안내에 머문다. 유아교육기관(어린이집·유치원)은 영유아라는 고위험군을 다수 보호하지만, 현장에서 "그래서 지금 무엇을 해야 하는가"를 역할별로 즉시 실행 가능한 행동으로 번역하는 도구가 없다. 본 제품은 그 **번역(Translation) 갭**을 메운다.

---

## 2. 문제 정의

| # | 문제 | 현장 영향 |
|---|---|---|
| P1 | 재난문자가 일반 시민 기준 → 유아기관 현장 행동으로 미번역 | 원장·교사가 매번 자체 판단, 누락·지연 발생 |
| P2 | 역할별(원장/담임/통학버스) 책임이 분화되지 않음 | "누가 무엇을" 불명확, 중복·공백 |
| P3 | 학부모 안내문 작성 부담 + 과도한 공포 표현 위험 | 안내 지연 또는 불안 조장 |
| P4 | 사후기록 작성 부담 | 조치 사실 입증·개선 누적 어려움 |
| P5 | 폭염 취약 유아(온열·호흡기·이동지원 등) 정보가 개인정보 우려로 활용 곤란 | 위험 대응이 일반화됨 |

→ **해결 방향**: 재난문자 + 기관정보 + 현재 운영상황(집계값 기반)을 입력하면, AI가 **역할별 체크리스트 / 학부모 안내문 / 사후기록 초안**을 JSON으로 생성. 개인식별정보는 수집·전송하지 않는다.

---

## 3. 핵심 사용자

| 역할 | 주요 니즈 | MVP 진입점 |
|---|---|---|
| **지자체 관리자** | 관할 기관 현황·우선순위 모니터링 | 관리자 대시보드 |
| **유아교육기관 원장** | 기관 전체 대응 의사결정·총괄 | 대응계획 생성 + 원장 체크리스트 |
| **담임교사** | 보육실 단위 즉시 실행 행동 | 담임 체크리스트 |
| **통학버스 담당자** | 차량 폭염 안전(탑승 전/하차) 행동 | 통학버스 체크리스트 |

> MVP는 복잡한 인증 대신 **시연용 역할 선택**으로 위 진입점을 구분한다.

---

## 4. MVP 목표

공모전 제출 전까지 **작동 가능한 웹 MVP**를 완성한다. 다음 흐름이 끊김 없이 시연되어야 한다.

```
역할 선택
  → 기관 등록
  → 폭염 대응 프로필 등록
  → 폭염 재난문자 입력(샘플 선택 또는 원문 붙여넣기)
  → 현재 기관 상황 선택(최대 3개)
  → AI 맞춤 대응계획 생성(JSON)
  → 원장·담임교사·통학버스 담당자별 체크리스트 표시
  → 학부모 안내문 생성(복사)
  → 사후기록 저장
  → (관리자) 대시보드에서 현황 확인
```

---

## 5. 기능 요구사항

### FR-1. 역할 선택 (6.1)
- 4개 역할(지자체 관리자/원장/담임교사/통학버스 담당자) 버튼 선택 또는 간단 로그인.
- MVP는 시연용 역할 선택을 기본으로 한다.

### FR-2. 기관 등록 (6.2)
입력 필드: 기관명, 기관유형(어린이집/유치원), 주소, 위도, 경도, 시도, 시군구, 행정동, 전체 유아 수, 영아 수, 유아 수, 교직원 수, 통학버스 운영 여부, 실외놀이터 여부, 냉방 가능 공간 수, 물 공급 가능 여부.

### FR-3. 폭염 대응 프로필 (6.3)
입력 필드: 온열취약 유아 수, 호흡기 주의 유아 수, 이동지원 필요 유아 수, 특별지원 필요 유아 수, 냉방기 정상 여부, 실내 대체활동 공간 여부, 정수기/물 공급 가능 여부, 체온계 보유 여부, 구급함 보유 여부, 차량 내부 온도계 보유 여부, 하원 대기 장소 유형(실내/그늘/실외/기타).
- **유아 이름·진단명·보호자 연락처 저장 금지. 민감정보는 집계값(숫자)으로만 저장.**

### FR-4. 재난문자 입력 (6.4)
- 샘플 폭염 재난문자 선택 / 실제 재난문자 원문 붙여넣기.
- 향후 행정안전부 긴급재난문자 API 연동 후보.

### FR-5. 현재 상황 선택 (6.5)
- 버튼형 UI, **최대 3개 선택**.
- 선택지: 실외놀이 전 / 실외놀이 중 / 산책·현장학습 예정 / 급식 전후 / 낮잠 시간 / 하원 준비 / 통학버스 탑승 전 / 냉방 이상 있음 / 온열증상 의심 유아 있음 / 특별 일정 없음 / 기타 직접 입력.

### FR-6. AI 대응계획 생성 (6.6)
AI는 다음을 **JSON**으로 생성: 재난문자 요약, 대응 우선순위, 반영된 근거 정보, 부족한 정보, 원장용 체크리스트, 담임교사용 체크리스트, 통학버스 담당자용 체크리스트, 학부모 안내문, 사후기록 초안, 응급 연락 안내, 공식기관 지시 우선 안내문.
- **JSON 파싱 실패 시 재시도 또는 샘플 결과 fallback 표시.**

### FR-7. 학부모 안내문 (6.7)
- 복사 버튼 제공. 과도한 공포 표현 금지. 기관이 이미 조치 중이라는 안정감 제공. 하원/물병/차량 대기/건강상태 확인 등 구체 행동 포함.

### FR-8. 사후기록 (6.8)
- 항목: 재난문자 확인 시각, 실외활동 조정 여부, 냉방 확인 여부, 유아 건강 이상 여부, 학부모 안내 여부, 통학버스 확인 여부, 조치 완료자, 특이사항, 개선 필요사항.

### FR-9. 관리자 대시보드 (6.9)
- 등록 기관 수, 오늘 생성된 대응계획 수, 대응 우선순위 높은 기관 수, 기관별 최근 대응계획 목록, 기관 상세 보기.

## 6. 비기능 요구사항

| 구분 | 요구사항 |
|---|---|
| **시연 안정성** | 네트워크/외부 API/AI 실패 시에도 **샘플 fallback**으로 전 흐름 동작. 데모 중단 금지. |
| **응답 시간** | AI 생성 목표 10초 이내, 초과 시 로딩 표시 + 타임아웃 후 fallback. |
| **개인정보·안전** | 10절 원칙 준수. AI API에 개인식별정보 미전송. |
| **접근성/모바일** | 현장 모바일 사용 가정. 큰 터치 타깃, 버튼형 UI, 반응형. |
| **신뢰성 표기** | 결과 화면에 안전 고지 문구 고정 노출. |
| **유지보수** | 재난유형 확장을 고려한 구조(폭염을 첫 케이스로 일반화 여지). |

### 개인정보 및 안전 원칙 (필수)
- 유아 이름 저장 금지 / 진단명 저장 금지 / 약물명 저장 금지 / 보호자 연락처 저장 금지.
- 취약 유아 정보는 숫자 집계값만 저장.
- AI API에는 개인식별정보를 전달하지 않음.
- AI는 의료 진단을 하지 않음.
- AI는 공식기관 지시보다 우선한다고 표현하지 않음.
- 결과 화면 고정 문구: **"공식 재난문자와 기관 입력정보를 바탕으로 한 대응지원 정보이며, 위급상황에서는 공식기관 지시와 119 안내를 우선한다."**

## 7. 제외 범위 (MVP 미구현)

모바일 앱 / 스마트폰 재난문자 자동 수신 / 카카오 알림톡 자동 발송 / 통학버스 노선 위험분석 / 이미지 분석 / 동영상 분석 / 유아별 건강정보 관리 / 실제 지자체 전체 기관 권한체계 / IoT 센서 연동.

추가 재난유형(집중호우·대설·화재·감염병·미세먼지)은 **확장 기능 문서**로만 정리.

## 8. 성공 기준

### 8.1 핵심 5원칙 (최우선)
1. 시연 흐름이 끊기지 않을 것.
2. 샘플 데이터 fallback이 있을 것.
3. AI 결과가 JSON으로 안정적으로 표시될 것.
4. 개인정보를 과도하게 수집하지 않을 것.
5. 재난문자·기관정보·현재상황이 AI 결과에 근거로 반영될 것.

### 8.2 측정 가능 기준
- [ ] 신규 사용자가 역할 선택→사후기록까지 **막힘 없이 1회 완주** 가능.
- [ ] 외부 API/AI 차단 환경에서도 fallback으로 전 화면 정상 동작.
- [ ] AI 출력이 정의된 JSON 스키마(04 문서)를 100% 충족, 파싱 실패 시 자동 fallback.
- [ ] DB·AI 전송 데이터에 개인식별정보 0건(스키마·프롬프트 차원에서 원천 차단).
- [ ] 결과 화면에 "재난문자 요약 + 반영 근거"가 항상 표시되어 입력→결과 연결이 드러남.

### 8.3 심사 데모 체크리스트
- [ ] 샘플 재난문자 1건으로 30초 내 역할별 체크리스트 생성 시연.
- [ ] 학부모 안내문 복사 버튼 동작 시연.
- [ ] 관리자 대시보드에서 우선순위 높은 기관 식별 시연.
- [ ] 안전 고지 문구 노출 확인.

---

## 9. 용어
- **대응계획(action plan)**: 하나의 재난문자 + 기관 상황 입력에 대해 AI가 생성한 결과물 묶음.
- **집계값**: 개인을 식별하지 않는 숫자 합계(예: 온열취약 유아 수).
- **fallback**: 외부 의존(AI/API/DB) 실패 시 미리 준비된 샘플 데이터로 흐름을 유지하는 메커니즘.
