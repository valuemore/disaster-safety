# 00_PRD — 재난안전MVP 제품 요구사항 문서

> 본 문서는 공모전(제품·서비스 개발 부문) 제출용 **작동형 웹 MVP**의 제품 요구사항을 정의한다.
> 완성형 상용 서비스가 아니라, **심사위원에게 핵심 시연 흐름을 안정적으로 보여주는 시제품**을 목표로 한다.
> 최종 갱신: **2026-06-16** (P8~P11 완료 반영 — 3유형·5역할 확장)

---

## 1. 제품 개요

- **프로젝트명**: 재난안전MVP
- **한 줄 설명**: 지자체 재난문자를 유아교육기관의 실제 운영상황에 맞춰 **원장·담임교사·통학버스 담당자·조리사/급식담당자·보건담당자별 실행 체크리스트**, **학부모 안내문**, **사후기록**으로 변환하는 AI 기반 재난대응 웹 MVP.
- **지원 재난유형**: **폭염(heatwave) · 집중호우(heavy_rain) · 감염병(infection)** 3종 구현.
- **지원 역할**: **원장(director) · 담임교사(homeroom_teacher) · 통학버스담당자(bus_manager) · 조리사/급식담당자(cook_or_food_service) · 보건담당자(health_manager)** 5종.
- **배경**: 재난문자는 일반 시민 대상의 포괄적 안내에 머문다. 유아교육기관(어린이집·유치원)은 영유아라는 고위험군을 다수 보호하지만, 현장에서 "그래서 지금 무엇을 해야 하는가"를 역할별로 즉시 실행 가능한 행동으로 번역하는 도구가 없다. 본 제품은 그 **번역(Translation) 갭**을 메운다.

---

## 2. 문제 정의

| # | 문제 | 현장 영향 |
|---|---|---|
| P1 | 재난문자가 일반 시민 기준 → 유아기관 현장 행동으로 미번역 | 원장·교사가 매번 자체 판단, 누락·지연 발생 |
| P2 | 역할별(원장/담임/통학버스/조리·급식/보건) 책임이 분화되지 않음 | "누가 무엇을" 불명확, 중복·공백 |
| P3 | 학부모 안내문 작성 부담 + 과도한 공포 표현 위험 | 안내 지연 또는 불안 조장 |
| P4 | 사후기록 작성 부담 | 조치 사실 입증·개선 누적 어려움 |
| P5 | 취약 유아(온열·호흡기·이동지원·특별지원 등) 정보가 개인정보 우려로 활용 곤란 | 위험 대응이 일반화됨 |
| P6 | 폭염 외 집중호우·감염병 상황에서 유아기관 전용 행동지침 부재 | 재난유형별 맞춤 대응 불가 |

→ **해결 방향**: 재난문자 + 기관정보 + 현재 운영상황(집계값 기반)을 입력하면, AI가 **역할별 체크리스트 / 학부모 안내문 / 사후기록 초안**을 JSON으로 생성. 개인식별정보는 수집·전송하지 않는다.

## 3. 핵심 사용자

| 역할 | 주요 니즈 | MVP 진입점 |
|---|---|---|
| **지자체 관리자** | 관할 기관 현황·우선순위 모니터링 | 관리자 대시보드 |
| **유아교육기관 원장** | 기관 전체 대응 의사결정·총괄 | 대응계획 생성 + 원장 체크리스트 |
| **담임교사** | 보육실 단위 즉시 실행 행동 | 담임 체크리스트 |
| **통학버스 담당자** | 차량 안전(탑승 전/운행 중/하차) 행동 | 통학버스 체크리스트 |
| **조리사/급식담당자** | 급식·식재료 안전, 조리실 위험 대응 | 조리 체크리스트 |
| **보건담당자** | 유증상 관찰·분리·보호자 연락·공식기관 안내 | 보건 체크리스트 |

> MVP는 복잡한 인증 대신 **시연용 역할 선택**으로 위 진입점을 구분한다.

## 4. MVP 목표

공모전 제출 전까지 **작동 가능한 웹 MVP**를 완성한다. 다음 흐름이 끊김 없이 시연되어야 한다(3개 시연 경로 모두).

```
역할 선택
  → 기관 등록 (급식·보건 인력 정보 포함)
  → 재난유형 선택 (폭염 / 집중호우 / 감염병)
  → 재난유형별 대응 프로필 등록
  → 재난문자 입력(샘플 선택 또는 원문 붙여넣기) [감염병은 선택]
  → 현재 기관 상황 선택(최대 3개, 재난유형별 상황 버튼)
  → AI 맞춤 대응계획 생성(JSON)
  → 원장·담임교사·통학버스·조리사/급식담당자·보건담당자별 체크리스트 표시
  → 학부모 안내문 생성(복사)
  → 사후기록 저장
  → (관리자) 대시보드에서 3유형별 현황 확인
```

## 5. 기능 요구사항

### FR-1. 역할 선택 (6.1)
- 5개 역할 + 지자체 관리자(총 6개) 버튼 선택.
- MVP는 시연용 역할 선택을 기본으로 한다.
- 기관 재원 수·학급 수·인력 정보를 기반으로 역할 활성화 추천(법적 단정 금지 — "확인이 필요합니다" 표현만 허용).

### FR-2. 기관 등록 (6.2)
입력 필드: 기관명, 기관유형(어린이집/유치원), 주소, 위도, 경도, 시도, 시군구, 행정동, 전체 유아 수, 영아 수, 유아 수, 교직원 수, 통학버스 운영 여부, 실외놀이터 여부, 냉방 가능 공간 수, 물 공급 가능 여부.

**급식·보건 인력 정보** (`staff_profile` JSONB): 1회 급식 인원, 급식 인력 보유 여부·수, 조리사 면허 보유 여부, 집단급식소 신고 여부, 보건 인력 보유 여부·유형·수 등.

### FR-3. 재난유형 선택 (신규)
- 지원 유형: **폭염 / 집중호우 / 감염병** 3종.
- 재난유형에 따라 이후 프로필 폼·상황 버튼·AI 입력이 자동 분기.

### FR-4. 재난유형별 대응 프로필 (6.3 확장)

**폭염 프로필 (`disaster_specific`):**
온열취약 유아 수, 호흡기 주의 유아 수, 이동지원 필요 유아 수, 특별지원 필요 유아 수, 냉방기 정상 여부, 실내 대체활동 공간 여부, 정수기/물 공급 여부, 차량 내부 온도계, 하원 대기 장소 유형.

**집중호우 프로필 (`disaster_specific`):**
저지대 위치 여부, 인근 하천·배수로·급경사지 인접 여부, 지하공간 보유 여부, 1층 출입구 구조, 하원 대기 장소, 실외 놀이터 위치, 대체 실내 대기공간 확보, 비상연락망 구비 여부, 침수·정전 시 대피공간, 이동지원 필요 유아 수.

**감염병 프로필 (`disaster_specific`):**
반별 유아 수(집계값), 영아반 여부, 특별지원 유아 수, 보건실/분리대기 공간, 손소독제·마스크·소독용품 보유, 보호자 연락 방식(앱/문자/전화/게시판), 감염병 대응 매뉴얼 보유, 등원중지 안내 템플릿 보유.

- **공통**: 체온계 보유 여부, 구급함 보유 여부, 실내 대체활동 공간 여부.
- **유아 이름·진단명·보호자 연락처 저장 금지. 민감정보는 집계값(숫자) 또는 boolean만.**

### FR-5. 재난문자 입력 (6.4 확장)
- 샘플 재난문자 선택(재난유형별 3종) / 실제 재난문자 원문 붙여넣기.
- 행안부 긴급재난문자 API 실시간 조회 탭(유형별 필터).
- **감염병**: 재난문자 없이 기관 내 상황 입력만으로도 AI 생성 가능(재난문자 옵션화).

### FR-6. 현재 상황 선택 (6.5 확장)
- 버튼형 UI, **최대 3개 선택**.
- 상황 버튼은 재난유형별로 다르게 표시(registry 기반 동적 로드):
  - **폭염**: 실외놀이 전·실외놀이 중·산책·현장학습 예정·급식 전후·낮잠 시간·하원 준비·통학버스 탑승 전·냉방 이상·온열증상 의심·조리실 온도 상승·급식 식재료 보관 확인 등 11개.
  - **집중호우**: 실외놀이 전·실외놀이 중·하원 준비·통학버스 운행 전·통학버스 운행 중·지하공간 사용 중·현장학습 예정·기관 주변 침수 우려·보호자 인계 지연 가능·정전·누수 발생·조리실 누수·급식 지연·낙상·불안 반응 유아 등 12개+.
  - **감염병**: 유증상/발열 유아·구토·설사·호흡기 증상·같은 반 유사증상 반복·보호자 연락 필요·등원중지 안내 필요·교실 소독·급식·화장실 위생·교직원 유증상·조리종사자 유증상·급식·식기 위생 강화 등 12개+.

### FR-7. AI 대응계획 생성 (6.6 확장)
AI는 다음을 **JSON**으로 생성: 재난유형, 재난문자 요약, 대응 우선순위, 반영된 근거 정보, 부족한 정보, **역할별 행동 배열(`role_based_actions[]`, 5역할 동적)**, 학부모 안내문, 사후기록 초안(`checked_items` JSONB), 응급 연락 안내, 공식기관 지시 우선 안내문, 안전 고지문.
- **JSON 파싱 실패 시 재시도 또는 재난유형별 샘플 결과 fallback 표시.**
- 비활성 역할은 `actions: []` 또는 "해당 없음"으로 처리(탭에서 read-only 표시).

### FR-8. 학부모 안내문 (6.7)
- 복사 버튼 제공. 과도한 공포 표현 금지. 기관이 이미 조치 중이라는 안정감 제공. 재난유형에 맞는 구체 행동 포함.

### FR-9. 사후기록 (6.8 확장)
- `checked_items` JSONB 기반 재난유형별 동적 체크 항목. 기존 폭염 boolean 5개는 레거시 호환 유지.
- 항목: 재난문자 확인 시각, 조치 완료자, 특이사항, 개선 필요사항 + 재난유형별 동적 체크 항목.

### FR-10. 관리자 대시보드 (6.9 확장)
- 등록 기관 수, 오늘 생성된 대응계획 수, **재난유형별 3종 건수 뱃지·막대**, 대응 우선순위 높은 기관 수, 기관별 최근 대응계획 목록, 기관 상세 보기, **기관 역할 지정 현황**.

## 6. 비기능 요구사항

| 구분 | 요구사항 |
|---|---|
| **시연 안정성** | 네트워크/외부 API/AI 실패 시에도 **샘플 fallback**으로 전 흐름 동작. 데모 중단 금지. 3유형 × 5역할 모든 경로 보장. |
| **응답 시간** | AI 생성 목표 10초 이내, 초과 시 로딩 표시 + 타임아웃 후 fallback. |
| **개인정보·안전** | 아래 §10 원칙 준수. AI API에 개인식별정보 미전송. |
| **접근성/모바일** | 현장 모바일 사용 가정. 큰 터치 타깃(min-h-[44px]), 버튼형 UI, 반응형. |
| **신뢰성 표기** | 결과 화면에 안전 고지 문구 고정 노출. |
| **유지보수** | registry 패턴으로 재난유형 추가 시 코드 분기 없이 설정 객체만 추가. |

### 개인정보 및 안전 원칙 (필수)
- 유아 이름 저장 금지 / 진단명 저장 금지 / 약물명 저장 금지 / 보호자 연락처 저장 금지.
- 취약 유아 정보는 숫자 집계값만 저장.
- AI API에는 개인식별정보를 전달하지 않음(화이트리스트 필드만 직렬화).
- AI는 의료 진단을 하지 않음. 공식기관·보건당국·의료기관 지시 우선 안내 필수.
- AI는 공식기관 지시보다 우선한다고 표현하지 않음.
- 결과 화면 고정 문구: **"공식 재난문자와 기관 입력정보를 바탕으로 한 대응지원 정보이며, 위급상황에서는 공식기관 지시와 119 안내를 우선합니다."**

### 감염병 추가 안전 원칙
- 질병명 확정·진단 금지. "확진"은 사용자 명시 입력 시에만 사용.
- 보건당국·의료기관·기관 매뉴얼 우선 안내 필수.
- 낙인·차별 우려 표현 금지. 증상 집계값만 사용.

### 역할 추천 원칙
- 기관 재원 수·학급 수 기반 역할 활성화 추천은 **"확인이 필요합니다"** 수준에 한정.
- "배치 의무입니다" 등 법적 단정 표현 금지.

## 7. 제외 범위 (MVP 미구현)

모바일 앱 / 스마트폰 재난문자 자동 수신 / 카카오 알림톡 자동 발송 / 통학버스 노선 위험분석 / 이미지 분석 / 동영상 분석 / 유아별 건강정보 관리 / 실제 지자체 전체 기관 권한체계 / IoT 센서 연동.

추가 재난유형(**대설·화재·미세먼지** 등)은 **확장 기능 문서**로만 정리. (집중호우·감염병은 P9·P10에서 구현 완료.)

## 8. 성공 기준

### 8.1 핵심 5원칙 (최우선)
1. 시연 흐름이 끊기지 않을 것 (3개 시연 경로 모두).
2. 샘플 데이터 fallback이 있을 것 (3유형 × 5역할).
3. AI 결과가 JSON으로 안정적으로 표시될 것.
4. 개인정보를 과도하게 수집하지 않을 것.
5. 재난문자·기관정보·현재상황이 AI 결과에 근거로 반영될 것.

### 8.2 측정 가능 기준
- [x] 신규 사용자가 역할 선택→사후기록까지 **막힘 없이 1회 완주** 가능 (경로 A/B/C).
- [x] 외부 API/AI 차단 환경에서도 fallback으로 전 화면 정상 동작 (smoke-test 44/44 PASS).
- [x] AI 출력이 정의된 JSON 스키마(04 문서)를 100% 충족, 파싱 실패 시 자동 fallback.
- [x] DB·AI 전송 데이터에 개인식별정보 0건(스키마·프롬프트 차원에서 원천 차단).
- [x] 결과 화면에 "재난문자 요약 + 반영 근거"가 항상 표시되어 입력→결과 연결이 드러남.

### 8.3 심사 데모 체크리스트 (3개 시연 경로)
- [x] **경로 A(폭염)**: 샘플 재난문자 1건으로 30초 내 5역할 체크리스트 생성 시연.
- [x] **경로 B(집중호우)**: 하원 준비·통학버스 운행 전·지하공간 사용 중 상황으로 5역할 체크리스트 생성 시연.
- [x] **경로 C(감염병)**: 발열 유아 발생 상황 입력만으로(재난문자 없이) 5역할 체크리스트 + 학부모 안내문 생성 시연.
- [x] 학부모 안내문 복사 버튼 동작 시연.
- [x] 관리자 대시보드에서 3유형별 우선순위 확인 시연.
- [x] 안전 고지 문구 노출 확인 (3유형 전부).

## 9. 용어
- **대응계획(action plan)**: 하나의 재난문자·상황 입력에 대해 AI가 생성한 결과물 묶음.
- **집계값**: 개인을 식별하지 않는 숫자 합계(예: 온열취약 유아 수, 발열 유아 수).
- **fallback**: 외부 의존(AI/API/DB) 실패 시 미리 준비된 샘플 데이터로 흐름을 유지하는 메커니즘.
- **registry**: 재난유형별 상황 버튼·프로필 필드·AI policy block 설정 객체(`lib/disaster/registry.ts`). 유형 추가 시 이 파일만 수정.
- **role_based_actions**: AI 출력의 역할별 행동 배열. 5역할 × 3유형 조합으로 생성.
