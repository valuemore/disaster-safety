# 01_DEVELOPMENT_PLAN — 개발 계획

> 목표: 공모전 제출용 작동형 웹 MVP를 **시연 안정성 최우선**으로 완성한다.
> 기술 스택: Next.js(App Router) + TypeScript + Tailwind CSS + shadcn/ui + Supabase PostgreSQL + Anthropic Claude API + Vercel 배포.
> 확정 결정: AI=**Anthropic Claude**, 인증=**시연용 역할 선택**, 공공 API=**가능한 많이 연동(전 기능 샘플 fallback 필수)**, 저장소=**Supabase 우선 + 샘플 fallback**.

---

## 1. 개발 단계 (Phase)

| Phase | 내용 | 산출물 | 데모 가치 |
|---|---|---|---|
| **P0 스캐폴딩** | Next.js+TS+Tailwind+shadcn 초기화, Supabase 연결, env, 공통 레이아웃, SafetyNotice 컴포넌트, fallback 토글 인프라 | 기동되는 빈 앱 | 기반 |
| **P1 기관/프로필** | `institutions`·`heatwave_profiles` 마이그레이션, 시드, 기관등록 폼, 폭염 프로필 폼 | 기관 CRUD | 입력 |
| **P2 재난문자/상황** | 샘플 재난문자 셋, 재난문자 입력(샘플/붙여넣기), 현재상황 선택(최대3) | 입력 단계 완성 | 입력 |
| **P3 AI 생성(핵심)** | AI 입력 빌더(개인정보 제거), Claude Server Route, zod 스키마 검증, 1회 재시도, **샘플 fallback**, 결과 JSON 렌더 | 동작하는 AI 결과 | ★최중요 |
| **P4 결과 활용** | 역할별 체크리스트 탭, 학부모 안내문(복사), 사후기록 폼·저장 | 결과 소비 완성 | 핵심 |
| **P5 대시보드** | 관리자 대시보드(집계·목록·기관 상세) | 모니터링 | 차별화 |
| **P6 공공 API(가산점)** | 주소·좌표 변환, 기상청 단기예보 우선 연동(+ 영향예보/재난문자/기관정보), 항상 fallback 동반 | 실데이터 연동 | 가산점 |
| **P7 시연 폴리시·배포** | fallback 강제 점검, 시드 정비, 데모 스크립트, 반응형 마감, Vercel 배포 | 제출본 | 마감 |

> 원칙: **P0→P4까지가 데모 본선**. P3가 완료되면 핵심 가치가 증명된다. P6는 시간이 남을 때 가산점으로 쌓되, 연동 실패가 데모를 깨지 않도록 fallback을 먼저 보장한다.

---

## 2. 작업 순서 (권장 시퀀스)

1. P0 스캐폴딩 → 앱 기동·레이아웃·SafetyNotice·env.
2. P1 DB 스키마 적용 + 시드 + 기관/프로필 폼.
3. P2 재난문자·상황 선택 입력.
4. **P3 AI 생성 라우트 + 검증 + fallback + 결과 렌더** (여기서 한 번 끝까지 흐름 연결).
5. P4 체크리스트/안내문/사후기록.
6. P5 대시보드.
7. P6 공공 API 연동(주소·좌표 → 단기예보 → 나머지).
8. P7 시연 폴리시·배포.

> 권고: P3 직후 **"역할선택→사후기록까지 1회 완주 스모크 테스트"**를 실시해 흐름 단절 지점을 조기에 제거한다.

## 3. 마일스톤

| 마일스톤 | 정의(Definition of Done) |
|---|---|
| **M1 핵심 흐름 동작** | 역할선택→기관등록→프로필→재난문자→상황선택→AI생성(또는 fallback)→결과 표시까지 끊김 없이 완주 |
| **M2 AI 안정화** | AI 출력이 JSON 스키마 100% 충족, 파싱 실패 시 자동 재시도→fallback. 개인정보 미전송 확인 |
| **M3 결과 활용 + 대시보드** | 역할별 체크리스트/학부모 안내문(복사)/사후기록 저장 + 관리자 대시보드 동작 |
| **M4 데모 준비완료** | 공공 API(최소 주소·좌표·단기예보) 연동 + 전면 fallback 검증 + Vercel 배포 + 데모 스크립트 완성 |

## 4. 예상 리스크 및 완화책

| 리스크 | 영향 | 완화책 |
|---|---|---|
| **AI JSON 파싱 실패** | 결과 화면 깨짐 | zod 스키마 강제 + JSON-only 프롬프트 + 1회 재시도 + 샘플 fallback(04 문서) |
| **공공 API 불안정/인증키 발급 지연** | 연동 미완 | API는 가산점으로 분리, 전부 Server Route + 샘플 응답 fallback. 키 미발급 시 자동 샘플 모드 |
| **Supabase RLS/네트워크 시연 충돌** | 저장·조회 실패 | service-role은 서버 라우트에서만, MVP RLS 단순화, DB 실패 시 in-memory 샘플 시드로 표시 |
| **시연장 네트워크 불안정** | 데모 중단 | `USE_SAMPLE_FALLBACK` 강제 토글로 오프라인 데모 모드 운영, 사전 시드 |
| **AI 응답 지연** | 체감 저하 | 로딩 UI + 타임아웃 후 fallback, 모델 기본 `claude-haiku-4-5` |
| **개인정보 유입 위험** | 원칙 위반 | 스키마에 PII 컬럼 부재 + AI 입력 빌더에서 화이트리스트 필드만 직렬화 + 자유텍스트 경고 |
| **일정 부족** | 미완 제출 | P0–P4 우선 고정, P5/P6는 가산점으로 후순위 |

## 5. 시연 준비 계획

- **시드 데이터**: 기관 3~5곳(어린이집·유치원 혼합, 시도/시군구 다양), 각 폭염 프로필, 샘플 재난문자 3종, 데모용 대응계획 1~2건.
- **샘플 재난문자 셋**: 경보/주의보/야외활동 자제 등 강도별 3종(02·04 문서와 공유).
- **데모 시나리오 스크립트**: (1) 원장 로그인→실외놀이 중+온열증상 의심 선택→AI 생성→체크리스트→학부모 안내문 복사, (2) 관리자 대시보드 우선순위 확인, (3) 사후기록 저장.
- **fallback 강제 토글**: `USE_SAMPLE_FALLBACK=true`로 외부 의존 전면 차단해도 동일 데모 가능함을 사전 검증.
- **리허설**: 제출 전 M1·데모 스크립트 2회 완주 리허설.
